# 12 - 项目起步与结构规划（Project Kickstart）

> 本章承接前几章（状态管理、导航逻辑、权限/认证），围绕 Canvas 上的项目/大作业说明，帮你把「页面与路由设计」真正落地成一个清晰的项目目录与文件骨架。

## 1. 项目骨架概览（初始化草稿）

在 Canvas 的项目说明和示例仓库中，通常会出现这样几类「必须有的」页面或模块：

- **首页（Home）**：
  - 角色：进入应用后的第一个落地点，用来介绍项目、给出主要功能的入口。
  - 对应前面章节：
    - 导航逻辑：作为 `/` 路由的目标页面；
    - 状态管理：通常展示部分全局状态的概览（例如任务数、收藏数等）。

- **主要功能页（Feature Pages，例如列表+详情）**：
  - 角色：承载应用的核心交互，比如任务列表、电影列表、帖子流等。
  - 对应前面章节：
    - 导航逻辑：对应如 `/tasks`、`/movies` 这类列表路由以及带参数的详情路由 `/tasks/:id`；
    - 状态管理：在这里读写大部分业务状态；
    - 权限/认证：某些功能页可能需要登录后才能访问。

- **辅助页面（Settings / About / Help 等）**：
  - 角色：提供设置、说明或调试入口，例如更改主题、查看项目说明、给出使用帮助。
  - 对应前面章节：
    - 导航逻辑：常作为导航栏中的次要入口；
    - 状态管理：部分全局偏好（如主题、语言）可以在这里修改。

- **错误与兜底页面（NotFound / Error Boundary）**：
  - 角色：在用户访问不存在的路径或发生未捕获错误时提供安全、友好的落点。
  - 对应前面章节：
    - 导航逻辑：承接 `path="*"` 兜底路由；
    - 权限/认证：在无权访问页面时给出清晰反馈。

后续节点会在这一节的基础上，结合 Canvas 的具体项目描述，把这些页面/模块进一步拆成实际的目录与文件命名建议（例如 `pages/`、`components/`、`routes/` 等），并画出一张从「页面与路由」到「文件结构」的对照表。

### 小结：先想清楚「有哪些房间」，再决定「怎么装修」

在正式画目录结构之前，可以先把这一节当成在画一张「房间列表」：

- 先列出这个应用里至少要有哪些房间（首页、主要功能区、设置/说明区、迷路时的安全出口）；
- 再回忆前面几章分别解决了哪些问题：
  - 导航逻辑：帮用户在不同房间之间走路；
  - 状态管理：记录房间里的东西、开关状态；
  - 权限/认证：决定谁能进哪些房间。

下一步要做的事情，就是把这些房间对应到具体的文件和目录上，让「路线图」真正变成一棵可以在编辑器里看到的项目树。

## 2. 从页面到目录：第一版项目树草稿

有了「房间列表」之后，可以先不看任何模板，直接用最朴素的方式把它翻译成一棵目录树草稿，例如：

```text
src/
  pages/
    HomePage.tsx          // 首页，对应 "/"
    FeatureListPage.tsx   // 主要功能列表页，对应如 "/items"
    FeatureDetailPage.tsx // 主要功能详情页，对应如 "/items/:id"
    SettingsPage.tsx      // 设置或偏好页面
    AboutPage.tsx         // 关于 / 帮助页面
    NotFoundPage.tsx      // 404 / 迷路页面
```

这棵树先只做一件事：**保证「每一个重要房间」在文件层面都有一个清晰的落点**，至于组件内部要不要再拆成子组件、样式文件怎么放，可以留给后面的章节慢慢细化。

在实际 Canvas 项目中，你看到的文件名可能更具体（例如 `MoviesListPage`、`TaskDetailPage`），但可以把它们都归纳到上面这几个角色里：

- HomePage：项目的门厅；
- 一组核心业务页面：列表 + 详情；
- 一组辅助/配置页面：Settings / About / Help；
- 一个兜底页面：NotFound。

接下来的节点会在这棵基础目录树上继续补充：

- 把导航栏、路由配置以及与认证/状态相关的「壳组件」放在什么位置；
- 如何在不打乱整体结构的前提下，为课程项目加入自己的额外页面或实验性功能。

> 提示：这一节只画出了最外层的 pages 目录。真正的项目往往还会有 `components/`、`routes/`、`layouts/` 等更多层级，后面会在不打乱这棵主树的前提下逐步补充，而不是一次性给出复杂模板。

## 3. 路由壳应该放在哪？（App 与 routes 的位置）

有了 `src/pages` 之后，还需要一个「壳」来把这些页面和路由系统串起来。典型的做法是：

```text
src/
  App.tsx           // 应用壳：挂载导航栏、全局布局、路由入口
  routes/
    AppRoutes.tsx   // 路由表配置：把路径映射到 pages 下的页面组件
  pages/
    ...             // 前一节列出的各个页面文件
```

一个常见的分工是：

- `App.tsx`：负责全局布局（比如导航栏、页脚、主题 Provider 等），内部引用 `AppRoutes`；
- `AppRoutes.tsx`：集中写路由表，把 `/`、`/items`、`/items/:id`、`*` 等路径分别指向 `pages/` 里的页面组件；
- `pages/`：只关心每个页面自己的内容，不关心路由机制细节。

这样做的好处是：

- 当你修改导航或路由配置时，主要在 `App.tsx` 和 `routes/` 下动手；
- 当你专注写页面内容时，只需要在 `pages/` 目录里增删文件，不会被路由细节打断。

当你后面在 Canvas 示例里看到各种不同的文件组织方式（例如：

- 有的直接在 `main.tsx` 里写路由；
- 有的把路由配置拆成多个文件；
- 有的甚至没有单独的 `routes/` 目录，只有一个很长的 `App.tsx`；

都可以先在脑中把它们「翻译」成上面的三层分工：

- 有一个壳：负责全局布局和挂载路由系统；
- 有一份路由表：负责路径到页面的映射；
- 有一组页面文件：负责各自的具体内容。

只要能在自己的项目树里找到这三类角色，大多数 Canvas 示例代码就不再显得「放得很奇怪」，而是可以对号入座。

## 4. 组件应该放在哪？（components/ 目录的组织思路）

在典型的 Canvas 项目示例或课程推荐结构中，除了 `pages/` 和 `routes/` 之外，还有一个频繁出现的目录：`src/components/`。它的定位可以用一句话概括：**放那些「在多个页面里都会用到」的通用部件**。

从示例仓库的描述中可以提炼出这样的组织原则：

- **页面专属组件** → 留在 `pages/` 或作为该页面的子模块：
  - 例如 `TaskDetailPage.tsx` 内部私用的 `TaskHeader`，不需要提升到全局；
  - 这类组件往往只服务于一个业务场景，迁移复用概率低。

- **跨页面复用组件** → 放到 `src/components/`：
  - 导航栏 `NavBar`（出现在所有页面顶部）；
  - 通用的按钮、卡片、加载状态 `LoadingSpinner`；
  - 布局外壳 `MainLayout`（包裹页面内容，提供统一的边距、背景）。

在课程示例中常见的 `components/` 结构通常长这样：

```text
src/
  components/
    NavBar/               // 导航栏组件
      NavBar.tsx
      NavBar.css          // 可选：组件级样式
    LoadingSpinner.tsx    // 纯展示组件，可单文件
    Button/
      Button.tsx
      Button.module.css   // 若使用 CSS Modules
```

关于样式文件的存放位置，不同示例仓库会有不同习惯：

- 有的示例把 `.css` 放在 `components/` 同级目录下；
- 有的示例使用 `ComponentName.module.css` 或 `styled-components`；
- 核心原则是：**让组件的「结构、样式、交互」三者尽量内聚**，方便在多个页面中直接引用而不需要额外配置路径。

> 💡 一句记忆法：当你写一个新组件时，先问自己——这个组件会不会在「两个或以上」的页面里出现？
> - 如果会 → 放进 `components/`；
> - 如果不会 → 先留在 `pages/` 内部，等真的需要复用时再迁移。

## 5. 从骨架到落地：最小可运行项目树速查

有了 pages/、routes/、components/ 的分工概念后，你可能会问：「Canvas 上的大作业到底要我交一个什么样的目录？」这一节把前面的内容收束成一张**最小可运行的项目树**，你可以把它当成脚手架的对照表。

### 根据 Canvas 要求调整目录

不同学期的 Canvas 项目说明可能会给出略有不同的目录建议，但核心要求通常是：

- **代码能跑**：有清晰的入口（`main.tsx` 或 `App.tsx`），能 `npm run dev` 启动；
- **页面完整**：实现了作业要求的所有路由和页面；
- **结构可读**：TA 能在几分钟内找到关键文件（路由表、页面组件、状态定义）。

如果 Canvas 给出的示例仓库结构和你手头的项目树不完全一致，不必慌张。可以用下面的「三层骨架」作为转换器：

| Canvas 可能用的目录名 | 对应本讲概念 | 你应该放什么 |
|:---------------------|:-----------|:-----------|
| `src/views/` 或 `src/screens/` | pages/ | 页面级组件，每个文件对应一个路由 |
| `src/router/` 或 `src/navigation/` | routes/ | 路由表配置，路径到组件的映射 |
| `src/ui/` 或 `src/shared/` | components/ | 跨页面复用的通用组件 |

### 最小可运行项目树（速查版）

以下是一个精简但完整的项目结构，足以支撑 Canvas 上的大多数课程项目：

```text
my-project/
├── public/                  // 静态资源（favicon、图片等）
├── src/
│   ├── main.tsx            // 应用入口：挂载 React 到 DOM
│   ├── App.tsx             // 应用壳：布局框架、全局 Provider
│   ├── routes/
│   │   └── AppRoutes.tsx   // 路由表：路径 → 页面组件
│   ├── pages/              // 页面目录：每个文件对应一个路由
│   │   ├── HomePage.tsx
│   │   ├── FeatureListPage.tsx
│   │   ├── FeatureDetailPage.tsx
│   │   ├── SettingsPage.tsx
│   │   ├── AboutPage.tsx
│   │   └── NotFoundPage.tsx
│   └── components/         // 通用组件：跨页面复用
│       ├── NavBar/
│       │   ├── NavBar.tsx
│       │   └── NavBar.css
│       └── LoadingSpinner.tsx
├── package.json
├── tsconfig.json
└── index.html
```

这张树的「最小」体现在：

- 没有过早拆分 `hooks/`、`utils/`、`types/` 等目录——它们可以在项目复杂度上升后逐步引入；
- 没有引入 `layouts/` 或 `contexts/` 等高级抽象——初学者可以先在 `App.tsx` 里集中管理；
- 保留了足够的扩展空间：当你需要加状态管理（Recoil）、认证逻辑（Firebase Auth）或更多工具函数时，都知道该在哪个层级插入。

### 一个实用的起步顺序

如果你面对 Canvas 项目感到不知从何开始，可以试试这个顺序：

1. **先画房间**（参考第1节）：列出作业要求的所有页面，给每个页面起一个 `.tsx` 文件名；
2. **再铺管道**（参考第3节）：在 `routes/AppRoutes.tsx` 里把路径和页面连起来，确保每个路径都能访问到；
3. **最后装修**（参考第4节）：把导航栏、加载状态等通用部件抽出来，放进 `components/`。

这个顺序的好处是**每一步都有可验证的产出**：先让路由能跑通（哪怕页面里只有占位文字），再逐步填充内容和美化。

接下来的节点会为这张项目树补充「避坑提醒」——在实际开发中，哪些目录决策容易导致后期重构，以及如何在一开始就避开它们。

## 6. ⚠️ 避坑提醒：项目结构的常见误区

在项目初期，有些「看起来很专业」的目录决策，往往会在后期带来不必要的麻烦。以下是 Canvas 作业和实际开发中经常出现的三类问题，以及简单的识别信号和应对策略。

### 坑 1：过早拆分目录（文件碎片化）

**典型表现**：
- 项目刚搭好 `src/` 就急着创建 `hooks/`、`utils/`、`types/`、`constants/`、`services/` 等空目录；
- 每个目录里只有一两个文件，甚至完全是空的；
- 想找一个功能时，需要在五六个目录之间跳转。

**为什么是个坑**：
- 目录是为了「把相关的东西放在一起」，而不是「把不同类型的东西分开」；
- 过早拆分会让文件之间的逻辑关系变得模糊，增加理解和维护成本。

**早期预警信号**：
- 你的 `utils/` 目录里只有一个 `formatDate.ts`；
- 你的 `types/` 目录和实际用到这些类型的组件文件相距甚远。

**应对策略**：
- **先平铺，后拆分**：一开始可以把工具函数、类型定义直接放在 `pages/` 或 `components/` 的同级目录下，等某个类别积累到 3-5 个文件时，再考虑独立目录；
- **按功能聚类，而非按类型聚类**：比如把「任务相关的类型、工具函数、API 调用」都放在 `features/tasks/` 下，而不是分散到 `types/`、`utils/`、`services/`。

### 坑 2：组件与页面界限模糊

**典型表现**：
- `components/` 目录里混进了大量只在某一个页面使用的组件；
- `pages/` 目录里的文件长达 500+ 行，包含大量子组件定义；
- 想复用一个组件时，发现它深度依赖某个页面的特定状态或数据。

**为什么是个坑**：
- 页面组件负责「布局和数据编排」，通用组件负责「展示和交互」；
- 界限模糊会导致复用困难，或者产生隐晦的循环依赖。

**早期预警信号**：
- 你发现自己在 `pages/HomePage.tsx` 里定义了 `HomePageHeader`、`HomePageCard`、`HomePageList` 等多个子组件；
- `components/UserCard.tsx` 的 props 里出现了只有 `ProfilePage` 才有的字段。

**应对策略**：
- **两页法则再确认**：在把组件放进 `components/` 之前，再次确认它是否会在两个或以上页面出现；
- **页面内子组件私有化**：如果某个子组件确实只在当前页面使用，可以把它定义在同一个文件底部（匿名函数），或者放在 `pages/HomePage/components/` 子目录下，而不是提升到全局 `components/`。

### 坑 3：路由表过于分散

**典型表现**：
- 每个 `pages/` 文件里都有自己的子路由定义；
- 路由配置被拆成 `routes/auth.ts`、`routes/main.ts`、`routes/admin.ts` 等多个文件，但没有统一的入口；
- 想查看「从首页到详情页有哪些路径」时，需要在五六个文件之间跳转。

**为什么是个坑**：
- 路由是应用的「导航系统」，应该一目了然；
- 分散的路由配置会让导航逻辑难以追踪，也容易产生路径冲突或遗漏。

**早期预警信号**：
- 你在 `App.tsx` 里看到了 `<Route>` 定义，同时又在 `pages/Dashboard.tsx` 里看到了 `<Routes>` 嵌套；
- 改了某个路径后，发现另一个地方还有硬编码的字符串没有同步更新。

**应对策略**：
- **单一入口原则**：把主要路由表集中在一个文件（如 `routes/AppRoutes.tsx`），让它成为路径的「唯一真相源」；
- **延迟拆分**：只有当某个模块的路由确实复杂到需要独立管理时（例如大型应用中的 admin 后台），才考虑拆分成 `routes/adminRoutes.tsx`，并在主路由表中显式引入。

### 小结：简单起步，渐进细化

项目结构不是一次性设计出来的，而是在开发过程中**逐步演化**的。一个好的经验法则是：

> **当你感到「找文件很麻烦」时，才是拆分目录的信号，而不是提前预判需要哪些目录。**

在 Canvas 作业的规模下，前面第5节给出的「最小可运行项目树」通常已经足够。等到你需要引入更多抽象（如全局状态、认证守卫、API 客户端）时，再按需扩展，比一开始就追求「完美的目录结构」要实用得多。

## 7. 本章小结

本章把前几章学到的「导航逻辑、状态管理、权限认证」映射到了实际的项目目录结构上。核心思路可以概括为四步：

1. **先画房间**（第1节）：列出应用必须具备的页面类型（首页、功能页、辅助页、兜底页），对应 Canvas 项目要求；
2. **再铺管道**（第2-3节）：用 `pages/` 存放页面，`routes/` 存放路由表，`App.tsx` 作为壳组件串联全局布局；
3. **最后装修**（第4节）：把跨页面复用的组件抽进 `components/`，遵循「两页法则」判断复用边界；
4. **避开深坑**（第6节）：警惕过早拆分目录、组件与页面界限模糊、路由表分散等常见误区，坚持「简单起步，渐进细化」。

这套「骨架→落地→避坑」的思维框架，不仅适用于 Canvas 的课程项目，也能延续到你未来的 React 项目实践中。

## 💡 TA 问答

**Q1: 我把所有组件都放在 `components/` 里，`pages/` 里只写路由跳转逻辑，这样有什么问题吗？**

**A1**: 这种做法会让 `components/` 迅速膨胀，失去「复用部件」的清晰定位。更棘手的是，当你试图复用某个看似通用的组件时，可能会发现它内部耦合了特定页面的业务逻辑（比如硬编码了某个页面的状态字段）。建议保留 `pages/` 作为「页面级容器」的专属领地，只有当组件确实跨页面复用时，才提升到 `components/`。

**Q2: Canvas 示例里的目录结构和我学的不一样，考试时应该按哪个来？**

**A2**: 评分标准通常关注「功能是否实现、代码是否可运行、结构是否清晰」，而不是「目录名是否和示例完全一致」。你可以用本章的「三层骨架」作为转换器——无论示例用 `views/` 还是 `screens/`，对应的都是「页面级组件」这一概念。只要在你的 README 或注释里简单说明目录分工，TA 能快速定位关键文件即可。
