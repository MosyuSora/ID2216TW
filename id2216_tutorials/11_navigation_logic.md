# 11 - 导航与路由逻辑

> 本章基于 Canvas 中与导航 / 路由相关的所有模块，总结如何在 React 项目中设计用户从「首页 → 列表 → 详情 → 404」的一整条路径。

## 1. 从 Canvas 示例项目看「用户旅程」

在 Canvas 的示例项目里，老师通常会画出这样一条典型路径：

1. 用户打开 **首页** (`/`)，看到课程或项目的整体介绍。
2. 点击导航栏中的 **某个功能入口**（例如 `Movies` / `Posts` / `Tasks`），URL 变成 `/movies` 或 `/tasks`，但页面并没有整页刷新，而是「局部区域」替换为对应列表页面。
3. 在列表中点击某一项，URL 变成 `/movies/42` 这一类「带参数」的地址，页面切换到详情视图。
4. 如果用户手动修改地址栏，输入了一个课程里没有配置的路径，例如 `/abc/xyz`，示例项目会显示一个「404」或「页面走丢了」的提示，并提供返回首页的按钮。

> 这一条路径就是本章的主线：**导航系统要帮用户在这些状态之间安全、清晰地来回移动**。

### 与 React Router 的对应关系（预告）

Canvas 示例里展示的这些路径，在代码层面通常会变成一张「路由表」：

```jsx
import { BrowserRouter, Routes, Route } from "react-router-dom";
import HomePage from "./HomePage";
import MovieListPage from "./MovieListPage";
import MovieDetailPage from "./MovieDetailPage";
import NotFoundPage from "./NotFoundPage";

export default function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/movies" element={<MovieListPage />} />
        <Route path="/movies/:movieId" element={<MovieDetailPage />} />
        <Route path="*" element={<NotFoundPage />} />
      </Routes>
    </BrowserRouter>
  );
}
```

在后续小节里，我们会一步步把 Canvas 中的例子拆开：

- 为什么在单页应用里，URL 变了页面却没有刷新？
- `:movieId` 这样的「占位参数」是怎么和课程实验里的 `id` 联系起来的？
- 顶部导航栏里的链接组件，如何和这张路由表配合？
- 404 页面应该如何设计，既满足课程要求，又对用户友好？

本节点只负责：**用文字重建 Canvas 示例项目里的那条「首页 → 列表 → 详情 → 404」路径，并给出一份对应的路由表示例**，不展开更多细节。后续节点会分别处理导航栏组件、参数化路由、404 设计等部分。

## 2. 顶部导航栏长什么样？（摘抄整理）

在 Canvas 的示例项目说明里，导航栏通常放在页面顶部，包含几类入口：

- 一个指向首页的入口，例如 `Home` 或课程项目的名字；
- 一到两个「功能页」入口，例如 `Movies`、`Tasks` 或 `Dashboard`；
- 可能还会有一个 `About` / `Help` / `Settings` 一类的辅助页面链接。

老师在讲解时经常会强调两件事：

1. **导航栏始终可见**：无论你当前在列表页还是详情页，导航栏都不会消失，方便用户随时回到首页或切换功能；
2. **导航项要和 URL 对得上**：如果导航里写的是 `Movies`，那点击之后的路径应该是 `/movies` 之类的有意义地址，而不是一堆看不懂的参数。

在代码里，这通常会变成一个单独的导航组件，例如：

```jsx
import { Link } from "react-router-dom";

export default function NavBar() {
  return (
    <nav>
      <Link to="/">Home</Link>
      <Link to="/movies">Movies</Link>
      <Link to="/about">About</Link>
    </nav>
  );
}
```

这个组件对应的，就是 Canvas 示例截图里那条横在最上方的导航条。后面的小节会继续从 Canvas 的实验和项目要求中，拆出「当前选中高亮」、「响应式布局」等细节。

## 3. 为什么 URL 变了页面却没「刷新」？（SPA 概念）

Canvas 在讲解示例项目时经常会提出一个问题：

> "你点了导航，地址栏变了，但是页面好像没有整页刷新，这到底发生了什么？"

这里其实是在引出 **单页应用（SPA, Single Page Application）** 的核心特点：

- 传统多页应用（MPA）：
  - 每点击一个链接，浏览器都会向服务器重新请求一整张 HTML；
  - 页面会出现明显的白屏 / 闪烁，开发者很难控制中间状态。
- 单页应用（SPA）：
  - 整个应用只在一开始下载一份 HTML + JS；
  - 之后的「跳转」由前端路由接管，只是**替换某个区域里显示的组件**，而不是整张页面；
  - URL 仍然会更新（方便书签和返回），但刷新行为被拦截成前端自己的逻辑。

React Router 正是在做这件事：

- 监听地址栏变化；
- 根据当前路径选择要渲染的组件；
- 保证像导航栏这种「全局框架」可以一直留在页面上，只替换中间内容区域。

在接下来的小节中，我们会继续结合 Canvas 的例子，从「用户点击导航 → URL 变化 → 路由表匹配 → 组件更新」这条链路，逐步拆开前端路由的工作原理。

## 4. 详情页为什么长成 `/movies/42`？（参数化路由场景）

Canvas 的实验和项目说明里，经常会让你做一件事：

> 在列表页点击某一项，例如某部电影 / 某篇帖子，把用户带到只展示这一项详情的页面。

在说明文档或截图中，你会看到类似这样的 URL：

- `/movies/42`
- `/posts/abc123`
- `/tasks/7`

这些地址有一个共同点：**后半段是「变化的」**。不同的电影 / 帖子 / 任务，会对应不同的 ID。为了在前端路由里表达这种「一部分固定、一部分可变」的规则，我们会在路由表中写出一个「带参数的路径」：

```jsx
<Route path="/movies/:movieId" element={<MovieDetailPage />} />
```

这里的 `:movieId` 就是一个**路径参数占位符**：

- Canvas 的说明里会告诉你，每一条电影记录都有一个 `id` 字段；
- 当用户点击某一条电影时，代码会把这条记录的 `id` 放到 URL 的最后一段；
- React Router 看到 `/movies/42` 这样的路径时，会用 `42` 这个值去填充 `:movieId`，然后把它交给 `MovieDetailPage` 组件去使用。

在后面的代码小节里，我们会进一步展示如何在组件里读取这个参数（比如通过 `useParams()`），以及如何根据它去加载或过滤对应的数据。这个节点先只做一件事：**把 Canvas 中「列表 → 详情」的场景翻译成参数化路由的路径形式**。

## 5. 走丢了怎么办？（404 / 迷路页面）

最后，还剩下用户旅程中的一块：**访问了一个不存在的路径**。

在 Canvas 的项目要求里，你通常会看到类似的说明：

> 如果用户输入了课程没有定义的 URL，应用应该显示一条友好的提示，并提供返回首页的按钮，而不是留在空白页面。

这就是我们常说的 **404 / Not Found 页面**。它的核心职责只有两件：

1. 告诉用户「这里没有内容」；
2. 给用户一条明确的退路，比如回到首页或者某个安全的起点。

在实际项目里，你可以设计很多有趣的 404 文案，例如：

- "你似乎走丢了，这一页不在本课程的路线图里。"
- "这里暂时没有内容，先带你回到首页继续学习。"

并在页面上提供一个清晰的按钮：

- 文案可以是：`返回首页` / `Back to Home`；
- 行为上，就是把用户带回到 `/` 或其他指定的安全路径。

在前面的路由表示例里，我们已经预留了一个兜底规则：

```jsx
<Route path="*" element={<NotFoundPage />} />
```

这一行的意思是：

- 当所有前面的具体路径（`/`、`/movies`、`/movies/:movieId` 等）都不匹配当前 URL 时；
- React Router 会落到这个最后的 `*` 路由上，渲染 `NotFoundPage` 组件；
- 由这个组件来负责展示友好的提示和「返回首页」按钮。

在接下来的节点里，我们可以结合 Canvas 的实际文案，进一步丰富这个 404 页面，让它和课程风格保持一致，但目前你只需要记住一点：**路由表的最后，总要有一个兜底的 NotFound 路由，帮用户「安全落地」**。

## 6. 从用户点击到组件渲染：把前面的点串起来（概念联通）

到目前为止，我们已经从 Canvas 的示例和说明里拆出了几块拼图：

- 导航栏上的链接，把用户从首页带到不同功能页；
- 路由表负责决定在当前 URL 下应该显示哪个页面组件；
- 参数化路径让「列表 → 详情」这样的场景可以通过 `/movies/42` 之类的地址来表达；
- 兜底的 404 路由在用户走到不存在的路径时接住他们。

把这些放在一起，可以看到 Canvas 想让你形成的那条「信号链」：

1. 用户在导航栏或页面中**点击一个链接**；
2. 浏览器地址栏中的 **URL 发生变化**（例如从 `/movies` 变成 `/movies/42`）；
3. 前端路由系统拦截这个变化，**在路由表里找到匹配的那一条规则**；
4. React 根据匹配结果，把对应的 **页面组件渲染到内容区域**，而导航栏等共同部分保持不变；
5. 如果没有任何规则匹配，就交给最后的 `*` 路由，显示 404 页面。

后续在 Canvas 的代码示例里，你会看到组件内部通过一些 Hook 或辅助函数「感知」当前 URL 和其中的参数，例如知道自己现在是在 `/movies/42`，还是在 `/movies`。在那之前，你可以先把这一章当作一张「解说图」：

- 上半部分说明用户是如何在不同页面之间移动；
- 下半部分说明这些移动背后，是怎样一套路由表和组件切换逻辑在工作。

只有先把这条链路的概念想清楚，后面看到具体 API 名字和代码时，才不会觉得是「记语法」，而是把已经理解的流程翻译成代码实现。

## 7. 组件怎么「感知」自己在哪？（为后续 Hook 做铺垫）

上一节我们把「从点击到渲染」这一整条信号链串了起来，但在 Canvas 的代码示例里，还有一个经常出现的问题：

> 组件本身怎么知道「我现在是在 `/movies` 列表页」，还是「我现在是在 `/movies/42` 某个详情页」？

Canvas 的讲解通常会通过两类例子来暗示这个问题：

- 在详情页组件里，根据 URL 里的某个 ID 去加载对应的数据条目；
- 在导航栏或侧边栏里，高亮当前所在页面对应的导航项。

这背后都依赖同一件事：**组件需要拿到「当前路径」以及其中的关键信息**，比如：

- 当前到底是 `/movies` 还是 `/movies/42`；
- 如果是详情页，路径里携带的是哪一个 `id`；
- 是否匹配到某个特定的子路由，用来决定菜单高亮。

在 React Router 里，这些信息最终会通过一组专门的工具暴露给组件使用（在 Canvas 示例中，你会看到一些以 `use...` 开头的辅助函数）。这一节先不写具体 API 名字，而是帮你把「需要什么信息」想清楚：

- 页面组件需要知道：我当前对应哪条路由规则？有没有路径参数需要读出来？
- 布局组件（例如导航栏）需要知道：当前 URL 属于哪一类页面，好决定高亮哪个入口；
- 某些保护性组件需要知道：当前要进入的路径是否在「允许访问的列表」之中。

把这些问题先放在脑子里，下一小节再用具体代码把它们一一对应起来，会比直接背诵函数名字更容易建立长期记忆。

### 小结表：不同组件各自需要的「路由视角」

| 组件角色   | 关心的路由信息                           | 典型用途                         |
|------------|------------------------------------------|----------------------------------|
| 页面组件   | 当前路径对应哪条路由、是否带有 `id` 参数 | 决定加载哪一条数据、显示哪个视图 |
| 布局/导航栏 | 当前 URL 属于哪类页面                   | 高亮当前菜单项、控制哪些入口可见 |
| 保护性组件 | 目标路径是否在允许列表、用户是否有权限   | 决定是放行、重定向还是拦截       |

这个表可以当作你在看 Canvas 示例代码时的「翻译表」：每当看到某个 Hook 正在读取路由信息，就问一句——它是帮哪一类组件完成上面的哪一个任务。

## 8. 常见路由相关问题场景一览（纯文字版）

在进入具体的 Canvas 代码示例之前，可以先用几种「常见提问方式」来检查自己是否抓住了本章的重点。这些问题在作业讨论区和 TA 答疑里经常出现：

1. **「为什么我点进详情页时，还是看到上一条数据？」**  
   - 场景：你从 `/movies` 列表点进第一条，看到的是 A 电影；返回列表再点第二条，URL 变成了 `/movies/2`，但页面上还是显示 A。  
   - 本质问题：详情页组件没有正确地「跟随当前 URL 中的 ID 去更新自己」，要么没有读到新的 ID，要么读到了却没有触发数据更新。

2. **「导航栏的高亮经常不对，要么全亮要么全不亮。」**  
   - 场景：当你从首页切换到 `/movies` 或者更深的 `/movies/42` 时，导航栏上的 `Movies` 有时候不高亮，有时候在别的页面也误高亮。  
   - 本质问题：导航栏对「当前 URL 是否命中某个入口」的判断条件不清晰，要么匹配得太宽（导致误高亮），要么匹配得太窄（导致该亮不亮）。

3. **「输入一个乱七八糟的地址时，应用要么白屏，要么跑到奇怪的页面。」**  
   - 场景：你在地址栏里键入 `/random/anything`，期望看到友好的 404 提示并能返回首页，结果却是空白页面或者报错。  
   - 本质问题：路由表里缺少一个可靠的「兜底规则」，或者兜底规则写了但没有正确指向一个能渲染提示信息的组件。

这三类问题背后都有一个共同点：

> 组件要做的事情没有错，错在「对当前路径的理解不准确」——要么没读对，要么没用好。

因此，在下一节真正展开 Canvas 示例中使用的那些工具之前，你可以先用这几种提问方式自测：

- 如果我在项目里遇到类似问题，我会先从「当前路径信息读得对不对」开始检查吗？
- 我能不能把问题翻译成「组件没看懂自己现在在哪、要展示谁」？

当你能用这些自然语言重新描述错误现象时，后面看到任何与路由相关的 API，都不会觉得是孤立的语法，而只是解决上述问题的不同「工具而已」。

## 9. 本章小结：从路径到体验（收束视角）

回到一开始那条「首页 → 列表 → 详情 → 404」的路线，现在可以用一句更完整的话来概括本章想做的事：

> 不只是教你写一张路由表，而是帮你建立一条从「用户怎样在应用里走路」到「代码怎样跟着 URL 做出反应」的映射关系。

如果把本章拆成三层，可以这样记：

1. **用户视角：我能去哪、会看到什么？**  
   - 首页 / 列表 / 详情 / 404，每一步都对应一个清晰的界面和下一步可选动作。
2. **路径视角：地址栏会长什么样？**  
   - `/`、`/movies`、`/movies/42` 到任意未匹配路径，分别表达「当前在整体介绍」「在某类内容的列表」「在某条具体内容详情」「在课程路线之外」。
3. **代码视角：组件要如何理解这些路径？**  
   - 路由表决定哪条路径对应哪个组件；
   - 组件通过读取当前路径和参数，决定展示谁、加载谁、放行谁。

后面在 Canvas 中看到的各种 Hook 和 API，可以都挂在这三层结构之下：有的负责「听 URL 的变化」、有的负责「读出其中的参数」、有的负责「据此决定要不要跳转或拦截」。

带着这张三层的思维导图，再回去看 Canvas 上的导航与路由代码，你可以一边对照本章的自然语言描述，一边问自己：

- 这行代码是在服务哪一层视角（用户 / 路径 / 代码）？
- 如果删掉它，用户在「走路」时会发生什么变化？

能回答出这些问题，说明你真正把导航与路由当成「用户体验的一部分」来理解了，而不是一堆必须记住的语法细节。
