# 第 10 章：Firebase Authentication

> 把用户登录想象成「实验室门禁系统」：先在管理处登记（注册），拿到带照片的门禁卡（账号），之后每次刷卡才能进实验室（访问受保护页面）。Firebase Auth 就是帮你托管这套门禁系统的云服务。

---

## 本章目标

- 理解 Firebase Authentication 提供的核心能力（注册 / 登录 / 退出登陆 / 持久化会话）。
- 能在 ID2216 课程项目中接入 Firebase Auth（基于 Web 前端）。
- 知道哪些逻辑应该放在前端、哪些交给 Firebase / 后端处理。
- 避免常见安全坑，比如在前端硬编码密钥、自己实现密码校验等。

---

## 1. 为什么需要认证？从「开放图书馆」到「门禁实验室」

先想象两个极端：

- **开放图书馆**：谁都能进，随便拿书、随便还书，系统只关心「有多少本书在架上」。
- **门禁实验室**：只有登记过的成员才能刷卡进入，不同人权限也不一样（学生 / 助教 / 老师）。

大部分现代 Web 应用，其实都更像「门禁实验室」而不是「开放图书馆」：

- 需要知道「你是谁」（Who are you）——否则无法记住你的设置、收藏、作业提交记录。
- 需要控制「你可以做什么」（What you can do）——比如只有登录学生才可以提交作业，只有老师可以改成绩。

在 ID2216 的课程项目里，一旦牵涉到：

- 保存用户的**个性化设置**（比如主题、语言偏好），
- 存储与用户绑定的**数据记录**（比如「我的任务列表」「我的书签」），
- 或者做「**只对部分人开放**」的页面（比如「课程项目管理后台」），

就离不开「认证（Authentication）」——验证用户身份的这一步。

Firebase Authentication 做的事情，就是帮你把「发卡、刷卡、验证卡片是否有效」这一整套流程托管到云端，让前端应用可以专注在**业务逻辑和界面展示**上，而不用自己造轮子实现账号系统。

> 和前几章的联动：
> - 你可以把「当前登录用户信息」看成一个全局状态（比如 Recoil 里的 `currentUserAtom`）。
> - 对登录状态变化的监听，会驱动 React 组件重新渲染，就像我们之前处理异步数据和 Suspense 一样。

## 2. Firebase Auth 基本概念速览

在真正写代码之前，先把几个高频名词对齐一下：

- **用户（User）**：
  - 在 Firebase Auth 里，对应一个有唯一 UID 的账号对象。
  - 账号下可以绑定多种登录方式（例如同一个人既可以用邮箱密码登录，也可以用 Google 登录）。
- **凭证（Credential）**：
  - 证明「这次登录请求是可信的」的一组信息，比如邮箱+密码、第三方登录返回的授权信息。
  - 在前端代码里，通常通过 Firebase 提供的 API 帮你构造，不需要自己手动拼接。 
- **ID Token / Refresh Token**：
  - ID Token：前端用来「带着身份去访问受保护资源」的一段加密字符串，短期有效。
  - Refresh Token：当 ID Token 过期时，用来静默刷新会话，让用户不用频繁重新登录。
- **认证状态监听（Auth State）**：
  - Firebase 会帮你在本地保存登录状态（例如存在 `localStorage` 或 `indexedDB` 中）。
  - 你可以注册监听函数，在用户登录 / 登出时自动触发回调，更新 React 状态。

典型支持的登录方式：

- 邮箱 + 密码（ID2216 项目中最常用、最稳定的方案）。
- 第三方提供商（Google、GitHub、Facebook 等）。
- 匿名登录（用于「先玩一玩，再注册」的体验，但课程里通常不强制要求）。

> 在本课程项目中，**推荐先从邮箱密码登录做起**：API 简单、便于调试，也足够支撑大部分作业场景。后续如果你有兴趣，可以再尝试接入 Google 登录作为扩展练习。

---

_本章后续会用到的一些关键 API（先记名字，后面章节会写示例）：_

- `createUserWithEmailAndPassword`：创建新用户（注册）。
- `signInWithEmailAndPassword`：使用邮箱+密码登录已有用户。
- `onAuthStateChanged`：监听当前登录状态变化。
- `signOut`：退出登录，清理本地会话。

## 3. 在 ID2216 项目中初始化 Firebase

这一节的目标，是把「云端的 Firebase 项目」和「你本地的 React 前端」连起来，让两边可以互相说话。

### 3.1 在 Firebase 控制台创建项目

- 登陆到 Firebase 控制台，新建一个项目（Project）。
- 给项目起一个和课程 / 作业相关的名字，方便区分不同环境。

### 3.2 为 Web 前端创建应用

- 在刚创建好的项目下，添加一个 **Web App**（带 `</>` 图标的那种）。
- 按向导填入应用昵称，完成后会看到一段配置代码（包含 `apiKey`、`authDomain` 等字段）。

### 3.3 在代码中初始化 SDK

- 在前端项目中新建一个专门放 Firebase 配置的文件（例如 `firebaseConfig.ts` 或 `firebase.ts`）。
- 把控制台里给出的配置对象粘进去，并通过 `initializeApp` / `getAuth` 等方法创建可复用的实例。

> ⚠️ 小提示（先记住，后面会展开）：
> - `apiKey` 在这里更像是一个「项目标识符」，**不是** 管理员密码，不能用它做安全边界。
> - 真正敏感的「管理级密钥」不应该出现在前端仓库里，更不要直接写进公开的 GitHub 仓库。

## 4. 实现邮箱密码注册 / 登录 / 退出

在大多数 ID2216 作业里，**邮箱 + 密码** 足够支撑完整的用户流程：

### 4.1 注册表单与基本校验

- 在前端实现一个简单的注册表单，收集邮箱和密码（必要时再加一个确认密码字段）。
- 在提交前做最基础的校验，例如「邮箱格式是否合理」「密码长度是否达到课程要求」。
- 校验通过后，调用 `createUserWithEmailAndPassword` 发送注册请求，并给出成功 / 失败反馈。

### 4.2 登录流程与错误提示

- 单独提供一个登录表单，让已有账号的用户输入邮箱和密码。
- 使用 `signInWithEmailAndPassword` 完成登录，根据返回的错误码（如密码错误、用户不存在）展示友好的提示信息。
- 登录成功后，把当前用户写入全局状态（例如 Recoil 的 `currentUserAtom`），并跳转到受保护页面或首页。

### 4.3 退出登录与状态重置

- 在导航栏或用户菜单中提供「退出登录」按钮。
- 点击时调用 `signOut`，并在回调中清理全局状态（例如把 `currentUserAtom` 重置为 `null`）。
- 可选：退出后将用户重定向到登录页或公开页面，避免停留在需要权限的路由上。

## 5. 保护「只对登录用户可见」的页面

当你已经有了「当前用户状态」之后，下一步就是**用好这个状态**：

### 5.1 基于路由的保护（Route Guard）

- 在路由配置层面包装受保护的路径，例如 `/dashboard`、`/profile`。
- 根据当前用户是否已登录，决定：
  - 已登录：正常渲染对应页面；
  - 未登录：重定向到登录页，或显示「请先登录」提示。

### 5.2 基于组件的保护壳（Protected Component）

- 实现一个通用的 `<Protected>` 组件，内部读取当前用户状态：
  - 有用户：渲染 `children`；
  - 无用户：渲染占位内容或触发路由跳转。
- 在需要保护的页面外面包一层 `<Protected>`，让权限逻辑集中在一个地方维护。

### 5.3 结合 Suspense 的加载状态

- 在使用 `onAuthStateChanged` 或异步获取当前用户信息时，短暂会有「尚未确定用户状态」的时间窗口。
- 可以配合第 7 章的 Suspense 思路，在用户状态未决时显示 Loading 占位，避免闪屏或错误跳转。

> ⚠️ 提醒：
> - 前端的路由保护更多是「用户体验层」的防护，**不能** 代替后端 / 数据库层的权限校验。
> - 真正敏感的数据仍然需要在后端检查 ID Token 是否有效，并基于服务端逻辑做授权判定。

## 6. ⚠️ 避坑：安全与隐私注意事项

- **不要在前端写管理密钥**：
  - 管理员级的服务账号密钥、后端 API 密钥等，一律只应保存在服务器或受保护的配置中，而不是 React 代码仓库。
- **不要把 Token 当「永久密码」长久保存**：
  - ID Token 有有效期，Refresh Token 也应该由 Firebase / 后端逻辑管理，避免自己造「永不过期的本地 Token」机制。
- **避免在公共设备上勾选持久登录**：
  - 在公共电脑 / 实验室机器上登录后，记得退出账号并清理浏览器会话，避免下一个使用者继承你的登录状态。

## 7. 💡 TA 问答：作业 / 项目里如何正确展示 Auth 使用

> 这一节是给「写报告 / 做答辩 / 回答 TA 问题」用的快速参考，不是背诵题库。

**Q1：在作业报告里，应该怎样说明自己使用了 Firebase Auth？**

- 建议写清楚 3 点：
  1. **用在哪里**：例如「登录页 + 受保护的任务列表页面」。
  2. **用到了哪些 API**：如 `createUserWithEmailAndPassword`、`signInWithEmailAndPassword`、`onAuthStateChanged`、`signOut` 等。
  3. **如何与 React 状态管理结合**：例如「登录成功后把当前用户写入 `currentUserAtom`，路由层根据用户状态决定是否允许访问某些页面」。
- 不需要在报告里贴所有代码，更重要的是说明**设计思路**和**与课程概念的对应关系**（比如「把登录状态当作全局 state」「用路由守卫保护页面」）。

**Q2：本地开发和部署到线上后，Auth 行为有点不一样，怎么在报告里解释？**

- 可以从以下角度说明：
  - 本地开发环境通常使用 `localhost` 域名，而线上环境使用真实域名，两者需要在 Firebase 控制台中正确配置授权域名。
  - 部署后，网络延迟 / HTTPS 配置可能影响首次加载和登录速度，可以在报告里简单提到你做了哪些处理（比如 Loading 占位、错误提示）。
  - 如果你为本地和线上分别配置了不同的 Firebase 项目，也可以说明你是如何区分这些环境配置的（如使用 `.env` 文件）。
- 可以给出一个简化的伪代码示例，展示你是如何区分环境并加载对应配置的，例如：

```ts
// firebaseConfig.ts
const firebaseConfig =
  process.env.NODE_ENV === "production"
    ? {
        apiKey: import.meta.env.VITE_FIREBASE_API_KEY_PROD,
        authDomain: import.meta.env.VITE_FIREBASE_AUTH_DOMAIN_PROD,
        // ...其他线上环境字段
      }
    : {
        apiKey: import.meta.env.VITE_FIREBASE_API_KEY_DEV,
        authDomain: import.meta.env.VITE_FIREBASE_AUTH_DOMAIN_DEV,
        // ...其他本地开发环境字段
      };

// 说明：
// 1. 不直接在代码里写死具体 key，而是从环境变量中读取；
// 2. 用 NODE_ENV 区分本地 / 线上配置，方便部署和调试；
// 3. 在报告中可以简要解释这一结构，而不需要贴出真实的 key。
```

- TA 更关心你是否意识到「环境差异」和「配置管理」，而不是每一个细节都做到完美。

**Q3：如何在作业中避免因为认证功能带来「学术诚信」方面的风险？**

- 避免把「同一个 Firebase 项目」硬编码给所有同学共用，尽量为自己的作业创建独立项目。
- 不要在前端公开展示可以访问其他同学数据的管理入口，即使只是「方便调试」。
- 如果你在团队项目中共享后端 / Firebase 项目，可以在报告里清楚说明分工和各自完成的部分，减少误解。
- 当你参考了课堂示例或公开教程中的 Auth 代码时，记得在报告中标明引用来源。
